# 百尺目标板块 → Zenith v2 移植评估

## 结论
**可以移植**。百尺目标板块与 Zenith 目标系统的数据模型高度同源，核心差异在于：
- 百尺多了一个 **`activeDays`（激活日期）** 概念，用于标记用户实际执行/努力的日期；
- 百尺的 UI 是“单目标 + 月历激活网格”，Zenith 是“多目标侧边列表 + 周条/小月历”。

移植工作主要集中在：**新增 `active_days` 字段 + 增加一个目标详情/月历视图**，逻辑改动不大。

---

## 1. 百尺目标板块现状

文件位置：`miniprogram/subpackages/goals/pages/dashboard/`

### 1.1 数据字段
```
title           目标名称
startValue      起始值
currentValue    当前值
targetValue     总目标
dailyTarget     每日增幅(%)
strategy        策略描述
startDate       目标开始日期
endDate         目标结束日期
activeDays      用户手动激活的日期数组 ['YYYY-MM-DD', ...]
```

### 1.2 显示结构
1. **目标概览卡片**：标题 + 进度百分比 + 进度条
2. **核心参数行**：起始值 / 当前值（可输入） / 总目标（可点击修改）
3. **每日增幅**：`- 5% +` 步进器 + 点击编辑
4. **计算结果行**：
   - 所需：从当前值到目标值需要的复利日数
   - 已激活：activeDays 长度
   - 还差/多出：|已激活 - 所需|
   - 预期值：按已激活天数复利计算后的最终值
5. **月历网格**：
   - 月份切换
   - 点击日期切换激活/取消激活
   - 激活日期显示红点 + 当日配额（按复利递增）

### 1.3 核心公式（已在百尺和 Zenith 中部分存在）
- 所需天数：`ceil(ln(target/current) / ln(1 + rate/100))`
- 预期值：`current * (1 + rate/100)^N`
- 第 k 天配额：`current * factor^k - current * factor^(k-1)`

## 2. Zenith 当前目标系统现状

文件位置：`frontend/src/components/AppLayout.tsx` / `frontend/src/features/GoalsView.tsx`

### 2.1 数据表（`backend/database.py`）
```sql
goals (
  id, title, start_value, target_value, current_value,
  daily_target, strategy, status, start_date, end_date
)
```
**缺少字段**：`active_days`。

### 2.2 当前显示（左侧面板 + GoalsView）
- **AppLayout 左侧**：多个目标折叠列表，显示进度条、当前/目标金额、日化、状态、快速更新余额
- **周条/小月历**：显示目标金额聚合（当前金额/日化目标/目标金额）
- **GoalsView**：独立目标列表页，卡片形式，展示进度和统计

### 2.3 已有功能（无需移植）
- 目标 CRUD
- 进度百分比计算
- 日化收益 / 剩余 / 在轨判断（服务端 `goal_get_stats`）
- 通过完成日程自动推进目标进度（`strategy: compound/linear`）
- 根据目标日自动反推 daily_target（`recalcDailyTarget`）

---

## 3. 可移植的部分

| 百尺功能 | 在 Zenith 中的对应 | 移植难度 |
|---------|------------------|---------|
| 目标概览卡片（标题+进度+参数） | GoalsView 的卡片已覆盖大部分 | 低 |
| 复利公式（所需天数、预期值） | 已存在于 `database.py`/`AppLayout.tsx` | 极低 |
| 每日增幅步进器 | 当前在表单里，可改在详情页内联 | 低 |
| 当前值直接输入 | GoalsView 已支持快速更新 | 无 |
| 月历激活网格 | **需要新增** | 中 |
| 激活日期数组 `activeDays` | **需要新增字段** | 中 |
| 激活日配额显示 | 依赖 `activeDays` 计算 | 中 |

---

## 4. 移植方案建议

### 方案 A：增强现有 GoalsView（推荐）
在 `GoalsView.tsx` 中新增一个“目标详情/月历”模式，点击某个目标后进入：
- 展示该目标的百尺式概览卡片 + 月历网格；
- 点击日期切换 `active_days`；
- 在左侧 AppLayout 保持多目标列表不变。

**优点**：不破坏现有布局，多目标与单目标深度管理分离。

### 方案 B：替换 AppLayout 左侧目标区为百尺式单目标视图
将左侧多目标列表改为“当前主目标 + 月历”形式。

**优点**：与百尺体验最接近。
**缺点**：Zenith 当前设计是多目标并行，单目标视图会弱化多目标能力。

---

## 5. 具体改动清单（若按方案 A）

1. **数据库**
   - 在 `goals` 表增加 `active_days TEXT DEFAULT '[]'`（JSON 字符串）。

2. **后端 API**（`backend/database.py` + `backend/app.py`）
   - 在 `_GOAL_COLUMNS` 中加入 `active_days`；
   - `goal_update` 支持 `active_days` 更新；
   - 可选：新增 `active_days` 计算接口（如按激活日返回配额数组）。

3. **前端**（`frontend/src/features/GoalsView.tsx`）
   - 新增目标详情视图：
     - 目标概览卡片（起始值/当前值/目标值/日化/所需天数/已激活/预期值/差额）；
     - 月历网格组件（可复用现有 `MiniMonth` 的日期计算逻辑）；
     - 点击日期 → 更新 `active_days`；
     - 根据激活日期排序计算每日配额。
   - 将百尺的 CSS 变量映射到 Zenith 的 Dracula 色板（`#50fa7b`, `#ff79c6`, `#ff5555` 等）。

4. **类型**
   - 在 `frontend/src/shared/api.ts` 的 `Goal` 接口中增加 `active_days?: string[]`。

---

## 6. 风险点

- **单目标 vs 多目标**：百尺默认单目标逻辑（`getCurrent`），Zenith 是多目标。移植时要在目标选择层面明确“当前查看哪个目标”。
- **activeDays 语义**：百尺中 `activeDays` 是用户手动标记的“努力日”，与日程完成无关；Zenith 已有“完成日程自动推进目标进度”机制，两者可并存，但需向用户解释清楚区别。
- **日期计算**：百尺用客户端本地日期，Zenith 服务端已有时区处理，确保统一用 `DEFAULT_TIMEZONE`（Asia/Shanghai）。

---

## 7. 总结

百尺的目标板块可以移植到 Zenith，且工作量可控（主要是一块新的目标详情 + 月历视图，以及一个 `active_days` 字段）。

建议采用**方案 A：在 GoalsView 中增加目标详情/月历模式**，既能保留 Zenith 多目标列表的现有能力，又能引入百尺的“激活日期 + 每日配额 + 预期值”视觉体验。
